QS 1: Discuss the scope of var, let, and const.

Ans: Variables that declared with 'var' are function-scoped. This means they are visible and accessible within the function in which they are defined. You can reassign and redeclare a var variable within the same scope.
-- Variables declared with 'let' and 'const' are block-scoped. They are visible and accessible only within the block in which they are defined, such as a function, loop, or conditional statement. You can reassign a let variable within the same block scope, but you cannot redeclare it in the same scope.
--Variables declared with const cannot be reassigned after they are assigned a value. However, for objects and arrays declared with const, their properties or elements can be mutated, but the variable itself cannot be assigned a new value.
--the scope of var is function-level, and it is hoisted. let and const have block-level scope, and they are not hoisted. let allows reassignment, while const enforces immutability for the variable itself but allows modifications to properties or elements of objects and arrays.
It's generally recommended to use 'let' and 'const' over 'var' in modern JavaScript development to avoid some common issues associated with var.

QS 2: Tell us the use cases of null and undefined.

Ans: Null is basically an assignment value. The variable which has been assigned as null contains no value and is empty. Undefined is when we declare a variable but do not assign a value to it, the variable becomes undefined.
** difference between null and undefined in JavaScript.
-Undefined-
1. Undefined means the variable has been declared, but its value has not been assigned.
2. The typeof() operator returns undefined for an undefined variable.
3. It returns NaN on performing arithmetic operations.
4. If there is no value assigned to the variable, it becomes undefined.

-Null-
1. Null means an empty value or a blank value.
2. The typeof() operator returns the type as an object for a variable whose value is assigned as null.
3. converts to 0 then perform the operation
4. If we assign null to a variable, it is an assignment value.

Use cases: Use undefined when you want to represent the absence of a value that hasn't been assigned yet, often for variables or object properties.
Use null when you want to explicitly indicate the absence of an object or value, typically as a placeholder for an object that will be assigned later or to represent the intentional absence of a value in your code.
It's important to note that both null and undefined are falsy values in JavaScript, which means they evaluate to false in a Boolean context. However, they have different semantic meanings, and it's essential to use them appropriately in your code to convey your intent accurately.

QS 3: What do you mean by REST API?

Ans: A REST API is an API that conforms to the design principles of the REST, or representational state transfer architectural style.

Some APIs, such as SOAP or XML-RPC, impose a strict framework on developers. But REST APIs can be developed using virtually any programming language and support a variety of data formats. The only requirement is that they align to the following six REST design principles - also known as architectural constraints:

1. Uniform interface. All API requests for the same resource should look the same, no matter where the request comes from. The REST API should ensure that the same piece of data, such as the name or email address of a user, belongs to only one uniform resource identifier (URI). Resources shouldn’t be too large but should contain every piece of information that the client might need.
2. Client-server decoupling. In REST API design, client and server applications must be completely independent of each other. The only information the client application should know is the URI of the requested resource; it can't interact with the server application in any other ways. Similarly, a server application shouldn't modify the client application other than passing it to the requested data via HTTP.
3. Statelessness. REST APIs are stateless, meaning that each request needs to include all the information necessary for processing it. In other words, REST APIs do not require any server-side sessions. Server applications aren’t allowed to store any data related to a client request.
4. Cacheability. When possible, resources should be cacheable on the client or server side. Server responses also need to contain information about whether caching is allowed for the delivered resource. The goal is to improve performance on the client side, while increasing scalability on the server side.
5. Layered system architecture. In REST APIs, the calls and responses go through different layers. As a rule of thumb, don’t assume that the client and server applications connect directly to each other. There may be a number of different intermediaries in the communication loop. REST APIs need to be designed so that neither the client nor the server can tell whether it communicates with the end application or an intermediary.
6. Code on demand (optional). REST APIs usually send static resources, but in certain cases, responses can also contain executable code (such as Java applets). In these cases, the code should only run on-demand.

RESTful APIs are widely used in web and mobile application development because for their simplicity, scalability, and compatibility with the HTTP protocol. They are commonly used for data retrieval, manipulation, and communication between client and server in a wide range of applications.
